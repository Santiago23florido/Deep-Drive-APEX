# Data and Runs

## Purpose of Run Artifacts

The repository stores and generates several kinds of data:

- Real run captures from the blue car.
- Simulation run captures from Gazebo.
- Logs and Docker tails.
- CSV traces for LiDAR, odometry, routes, local paths, and drive bridge status.
- JSON metadata and status files.
- Maps and point clouds.
- Diagnostic reports.

These artifacts are used for debugging, evaluation, mapping, replay, and post-run analysis.

## Main Data Locations

| Path | Purpose | Classification |
| --- | --- | --- |
| `APEX/apex_recognition_tour/` | Real recognition-tour run captures. | Current APEX data |
| `APEX/apex_curve_track/` | Real curve-tracking captures. | Current APEX data |
| `APEX/apex_forward_raw/` | Raw or forward-motion capture artifacts. | Current APEX data |
| `APEX/apex_static_curve/` | Static curve scan/capture artifacts. | Current APEX data |
| `APEX/analysis/` | Analysis outputs and derived artifacts. | Current APEX analysis |
| `APEX/.apex_runtime/` | Runtime status files such as session manager status. | Runtime state |
| `src/rc_sim_description/data/runs/` | Simulation run artifacts where enabled. | Simulation data |
| `recognition_tour_diagnostic_report.md` | Local diagnostic report for a recognition-tour run. | Diagnostic context |

Some run folders may be large. Treat them as experimental artifacts unless a specific run is promoted to a baseline dataset.

Local virtual environments, runtime PID/status folders, generated logs, and legacy simulator log exports are intentionally not part of the repository source set. Recreate Python environments locally, and let APEX regenerate `APEX/.apex_runtime/` and `APEX/ros2_ws/logs/` during execution.

## Recognition-Tour Run Structure

The real recognition-tour capture script writes under:

```text
APEX/apex_recognition_tour/<run_id>/
```

Typical files include:

| File | Meaning |
| --- | --- |
| `capture_meta.json` | Capture configuration, feature flags, and run metadata. |
| `recognition_tour_record.log` | Text log from the capture process. |
| `docker_tail.log` | Tail of container logs captured during the run. |
| `lidar_points.csv` | Exported LiDAR point data. |
| `recognition_tour_summary.*` | Summary output generated by analysis tools, when present. |
| `recognition_tour_status.*` | Planner/tracker status history, when present. |
| `recognition_tour_route.*` | Route points or route summary, when present. |
| `recognition_tour_trajectory.*` | Fused or tracked trajectory output, when present. |
| `drive_bridge_status.*` | Actuation bridge status history, when present. |
| `analysis_recognition_tour/` | Derived analysis and diagnostic artifacts, when present. |

Exact filenames vary by script version and run mode.

## Curve-Tracking Run Structure

Curve-track captures are usually written under:

```text
APEX/apex_curve_track/<run_id>/
```

Typical contents:

- Path planner outputs.
- Tracker status.
- Drive bridge status.
- Fused odometry.
- LiDAR scan snapshots or point exports.
- Summary and diagnostic files.

These runs support the curve-entry planner/tracker path rather than the recognition-tour planner/tracker path.

## Raw and Static Captures

| Capture family | Typical purpose |
| --- | --- |
| Forward/raw capture | Sensor-fusion experiments, raw IMU/LiDAR validation, PWM trace collection. |
| Static curve capture | Stationary LiDAR geometry and curve detection validation. |
| Rect/sensor-fusion capture | Movement and sensor-fusion development with additional event traces. |

The exact capture scripts live under:

```text
APEX/tools/capture/
```

## Runtime Status

The APEX session manager can write runtime status under:

```text
APEX/.apex_runtime/
```

The parameter file references a status path similar to:

```text
/work/repo/APEX/.apex_runtime/real_session/status.json
```

This path is inside the Docker container view of the repository. On the host, it corresponds to the mounted repository under `APEX/.apex_runtime`.

`APEX/.apex_runtime/` is generated runtime state. It should be ignored by Git and removed between sessions when stale PID/status files cause confusion.

## Simulation Run Data

Simulation recording tools in `rc_sim_description` can write run data under:

```text
src/rc_sim_description/data/runs/
```

Simulation artifacts may include:

- Simulated LiDAR and IMU data.
- Fused odometry.
- Ground-truth odometry.
- Ground-truth path.
- Ground-truth map points.
- Similarity or offline refinement metrics.

The simulation recorder is mainly useful for comparing estimated behavior against Gazebo ground truth.

## How to Interpret Run Data

Use this order when debugging a failed run:

1. Read `capture_meta.json` to understand feature flags and parameters.
2. Check planner status for path generation failures.
3. Check tracker status for stale path, abort, or arm-state failures.
4. Check fusion status and odometry for localization instability.
5. Check LiDAR point data for scan dropouts or incorrect mounting assumptions.
6. Check drive bridge status for clamps, disabled outputs, or backend errors.
7. Check `docker_tail.log` for exceptions and device errors.

## Diagnostic Reports

The repository may contain diagnostic Markdown files generated from a specific run. These reports are valuable context but should not automatically be treated as canonical documentation.

For example, the active `recognition_tour_diagnostic_report.md` describes one recognition-tour run where the final status was a path-loss abort and highlights possible planner/tracker synchronization and steering-limit issues. That report is useful for limitations and debugging, but the source code and current scripts remain the source of truth for architecture.

## Data Hygiene Recommendations

- Keep representative runs small and named clearly.
- Move large experimental runs out of the repository if they are not needed for reproducibility.
- Store run metadata with every capture.
- Preserve raw logs when diagnosing behavior changes.
- Avoid mixing real and simulated captures in the same output folder.
- Include the script command and git commit in run metadata where possible.
